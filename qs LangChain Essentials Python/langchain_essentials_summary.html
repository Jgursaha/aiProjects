Core Architecture: The ReAct Agent

The centerpiece of this course is the ReAct (Reasoning and Acting) Agent, built on top of LangGraph. This architecture synergizes reasoning and acting in language models through a continuous execution loop.

Request
↓
Model (Reason)
⤡ ⤢
Tools (Action & Observation)
↓
Result
  • Persistence: Built-in graph memory to remember across threads.
  • Interrupts: Enables Human-in-the-Loop workflows by pausing execution.
  • Tracing: Seamless integration with LangSmith for observability.

System State & Message Flow

In LangGraph, the agent's memory is simply a continuously growing list of messages appended to a shared scratchpad. State: "Messages" : [list of messages]

  • SystemMessage: "You are a helpful assistant"
  • HumanMessage: "Check my calendar"
  • AIMessage: (tool_call: calendar('schedule'))
  • ToolMessage: "Busy today"
  • AIMessage: "You are busy today"

The Middleware Lifecycle

Middleware allows you to insert specific code at key points in the ReAct loop to modify inputs, outputs, or behavior dynamically.

before_agent : Setup (files, connections)
↓
before_model : Summarization, guardrails
↓
wrap_model_call / wrap_tool_call : Dynamic prompts, retries, caching
↓
after_model : Output guardrails, Human-in-the-loop interrupts
↓
after_agent : Teardown

📓 Notebook Concept Breakdown

1. The Fast Agent (L1_FastAgent)

Concept: A ReAct agent is built with create_agent. It acts as the orchestration engine connecting your user requests, a language model (for reasoning), and your database/tools (for acting and observing).

  • Goal: Build an agent capable of exploring a SQLite database (Chinook.db) without prior schema knowledge.
  • Key Takeaway: The agent runs in a loop. If a SQL tool returns an error, the agent reads the error observation, self-corrects, queries the table names, and tries again.

2. Models and Messages (L2_Models_And_Messages)

Concept: Elements of the system communicate exclusively by appending message objects to a persistent State array.

  • Messages as Circulation: Information flows through standard message types: SystemMessage, HumanMessage, AIMessage, and ToolMessage.
  • Scratchpad: Messages aren't passed node-to-node; they append to a persistent scratchpad shared across the graph.

3. Streaming (L3_Streaming)

Concept: Since agentic loops can be slow, streaming allows you to emit data token-by-token or step-by-step directly to the user to support highly interactive applications.

  • Solutions (Stream Modes):
    • values: Emits state updates after every step.
    • messages: Streams the LLM output token-by-token.
    • custom: Allows emitting user-defined data streams from inside tools.

4. Tools (L4_Tools)

Concept: Tools provide the "Action" part of ReAct. The Reasoning Node uses the tool's python description to decide when and how to call the tool, and the result acts as the "Observation".

  • Action Generators: Standard Python functions wrapped in a @tool decorator.
  • Optimization: Setting parse_docstring=True explicitly parses Google-style docstrings into strict metadata schemas for the LLM.

5. Tools with MCP (L5_Tools_w_MCP)

Concept: You don't have to write all tools in Python. The Model Context Protocol allows the agent to reach out to an external server (e.g. running on Node.js) to execute tools.

  • Model Context Protocol: An open standard interface.
  • Usage: Utilized MultiServerMCPClient to connect to a local npx server, loading external tools directly into the LangChain agent.

6. Memory (L6_Memory)

Concept: Without memory, the agent forgets previous messages (like "What did I buy?"). By utilizing a Checkpointer and a Thread ID, the agent gains long-term or session-based memory.

  • Context: RuntimeContext is used to inject static dependencies (like DB connections) into tools.
  • Short-Term Memory: Fix amnesia by adding MemorySaver() (checkpointing) and passing a thread_id in the config.

7. Structured Output (L7_StructuredOutput)

Concept: Forcing the LLM to format its response as strict JSON so it can be parsed by subsequent computer systems (like extracting exact Name and Email strings).

  • Usage: By providing a response_format (like a Pydantic Model or a TypedDict), the agent outputs validated, structured data instead of raw text.

8. Dynamic Prompting (L8_Dynamic_Prompt)

Concept: Prompts must cover all phases and conditions of a task, but static prompts get too large. We can dynamically choose and inject the correct System Prompt at runtime using middleware.

  • Dynamic Prompts: Use middleware functions to conditionally build and swap System Prompts on the fly based on runtime context (e.g. table access limits).

9. Human in the Loop (HITL) (L9_HITL)

Concept: Some tool actions are too dangerous to run automatically. We can inject an interrupt hook into the loop to pause execution until a human reviews the action.

  • Safety & Control: Adding an interrupt breakpoint to the agent via after_model.
  • Usage: Pausing the graph to require manual human approval, rejection, or edits before executing sensitive tool calls like execute_sql.
